# Context
It is important to have a CLAUDE.md file at the root of the project (or computer user) to give it context. 

### Memory types:
- **Project memory**: `./CLAUDE.md`
- **User memory**: `~/.claude/CLAUDE.md`

### Filess, Folders, images
Add files/folders to the chat context by typing the @ symbol. When you have it, press TAB to add it.
- Shortcut: `Cmd + Option + K`

You can also refer files and folders from the CLAUDE.md file.

# Slash Command

## Claude Code Slash Commands

Slash commands are shortcuts you type during a conversation to trigger specific actions — things like `/clear` to reset context, `/model` to switch models, or `/cost` to check token usage. The full list changes as Claude Code evolves, so it's worth browsing the official docs to know what's available. [See the docs.](https://docs.anthropic.com/en/docs/claude-code/slash-commands)

### /init

Scans your project and generates a CLAUDE.md file with a summary of your codebase. This is a useful starting point, but you should review the output and trim it down — remove files, folders, or details that aren't essential. Keeping your CLAUDE.md focused prevents it from consuming unnecessary tokens on every conversation.

# Context 

## CLAUDE\.md (or AGENT\.md for other coding tools)
Keep your CLAUDE.md lean but intentional. Remove auto-generated noise, but add explicit guidance — your project's conventions, architectural principles, common patterns, or things you always want the AI to remember. You can also reference other files directly (like `package.json` or `requirements.txt`) instead of duplicating their contents. This lets Claude Code see your actual up to date dependencies.

You can maintain CLAUDE.md files at two levels: a user-level one in `~/.claude/CLAUDE.md` for guidance that applies to all your projects, and project-level ones in `./CLAUDE.md` for project-specific conventions and practices. 

## Automoatic Memory

The agent will build it's own memory over time. It might be nice to have that for some usecases, but for some other ones, it might not be useful. To access it, use the `/memory` command.

### Steering the agent

The Agent file is a good place to steer the agent for coding preferences or specific tasks that you want the agent to always have in mind, like making sure to always write tests.

#### Example for a coding preference

when you have a function with more than one parameter, use an object parameters instead of positional parameters:

```ts
// BAD
const addUserToPost = (userId: string, postId: string) => {};

// GOOD
const addUserToPost = (opts: { userId: string, postId: string }) => {};
```


## Adding Files, Folders, and Images as Context

There are several ways to get files into your conversation:

- **Automatically**: The currently open file and any selected text are included by default
- **Using @**: Type `@` followed by a filename or folder name to explicitly add it
- **Keyboard shortcut**: Press `Cmd+Option+K` (Mac) to open the file picker and add multiple items at once
- **Copy and paste**: Paste file or image content directly into the prompt

## Context Window

As your conversation grows, so does your context window. You can check your current usage with `/context`. When context gets too full, the model's effectiveness can degrade.

There are a few strategies to manage this:

- **Start fresh**: Begin a new session after completing major features or milestones
- **Jump back**: Use `/rewind` to return to an earlier point in your session and continue from there
- **Summarize**: Use `/compact` to generate a markdown summary of your conversation, which you can save and reference in a new session without carrying the full history forward 

# Tools

Under the hood, Claude Code uses tools to perform tasks. Because tools can do pretty important tasks like reading, editing, creating, deleting files (and more), it will ask you for permission before performing a sensitive task. By default, most of the tools used by Claude Code will require permissions before performing the task. Sometime, you might want to allow for certain tools to be called without granting permission everytime. Or, you might want to prevent certain actions (like pushing code to git). 

In a chat, you'll be asked to:
1. allow it 
2. Allow it all during the session
3. Refuse 

You can also run the prompt on "Accept edits on" mode. This will automaticelly accept everything. You can access it by pressing `SHIFT+TAB` to select the mode. 

## Permission Settings

You can configure tool permissions at three levels, each stored in a different file:

| Level | File | Shared via Git? |
|---|---|---|
| User | `~/.claude/settings.json` | No — applies to all your projects |
| Project | `.claude/settings.json` | Yes — shared with your team |
| Project (local) | `.claude/settings.local.json` | No — your personal overrides |

Each tool can be set to one of three states:

- **allow**: Claude runs the tool automatically without asking
- **ask**: Claude asks for your approval each time (the default)
- **deny**: Claude is never allowed to use the tool

### Specifiers

Some tools, like `Bash`, support specifiers to give fine-grained control over exactly which commands are permitted. The format is `ToolName(command:*)`. For example, `Bash(git commit:*)` allows git commit commands without prompting, but still asks for any other Bash command. The `*` is a wildcard that matches any arguments passed to that command — so you can whitelist specific safe operations instead of blanket-allowing all Bash access.

### Managing Permissions

Use `/permissions` to open the permissions UI and review or edit your current settings. When Claude asks for permission mid-conversation, you can:

1. **Allow once** — approve just this instance
2. **Allow always** — approve and save to `.claude/settings.local.json` so it never asks again for this project
3. **Deny** — block the action

# Running Bash Commands Directly

You can run a shell command from the Claude Code prompt by prefixing it with `!`. For example:

```
! npm test
```

The command runs in your terminal and its output is fed back into the conversation, so Claude can read it and respond. This is handy for debugging — run your tests, share the failure output, and ask Claude to fix it in one step.

Use it sparingly though. Large command outputs (build logs, verbose test suites) will consume a significant portion of your context window. Prefer it for short, targeted commands where the output is directly relevant to what you're asking Claude to do.

# Hooks

Hooks are shell commands that Claude Code runs automatically at specific points in its lifecycle. They let you enforce standards, automate follow-up tasks, or add guardrails — without having to ask Claude to do it each time.

Hooks are configured in `.claude/settings.json` (project) or `~/.claude/settings.json` (user) under a `hooks` key. You can also manage them interactively with the `/hooks` command, which opens a UI to add, edit, or remove hooks without touching the JSON directly.

## Lifecycle Events

| Event | When it fires |
|---|---|
| `PreToolUse` | Before Claude calls a tool (e.g., before editing a file) |
| `PostToolUse` | After Claude calls a tool (e.g., after writing a file) |
| `Notification` | When Claude sends a notification |
| `Stop` | When Claude finishes its response |

Each hook targets a specific tool via a `matcher` (a regex pattern) and runs a shell `command`.

## Example Configuration

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npm run lint --fix"
          }
        ]
      }
    ]
  }
}
```

## When to Use Hooks

**Lint or format after edits** — automatically run ESLint, Prettier, or Black every time Claude modifies a file, so the code is always clean without reminding it.

**Type-check after changes** — run `tsc --noEmit` after TypeScript edits to catch type errors immediately.

**Run tests after writes** — trigger your test suite after Claude creates or edits a file, feeding results back into the conversation.

**Desktop notifications** — use the `Stop` event to send a system notification when Claude finishes a long task, so you don't have to watch the terminal.

**Enforce guardrails** — use `PreToolUse` with a `Bash` matcher to block dangerous commands (e.g., `git push --force` or `DROP TABLE`) before they execute.

**Regenerate derived files** — after Claude edits an API schema or interface file, automatically regenerate TypeScript types, GraphQL bindings, or mock data.

## Example

### Simple example

When a hook fires, Claude Code passes a JSON payload to your hook script via **stdin**. A quick way to inspect that data is to pipe stdin through `jq` and write the result to a file:

```bash
jq . > tool-data.json
```

Save that as a script (e.g. `hook-debug.sh`), make it executable, and point your hook's `command` at it. Each time the hook fires, the pretty-printed JSON will be written to `tool-data.json` in the current directory.

You'll need `jq` installed — on macOS: `brew install jq`.
