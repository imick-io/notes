# Structure

Planning a feature before implementing it forces you to break the work into clear pieces to surface ambiguities before they become bugs. This is always valuable, but it matters even more when working with an LLM: if you aren't aligned on what's being built before the session starts, you'll discover the misalignment halfway through an implementation that's already hard to unwind.

The most effective approach is to create a **PRD or spec file** for the feature first. Ask Claude to produce a structured document — scope, requirements, constraints, open questions, acceptance criteria — before a single line of code is written. You review it, correct any misunderstanding, and hand it back as the source of truth. Implementation becomes execution against a shared plan rather than a negotiation mid-flight.

With the PRD in place, you then create a plan — the step-by-step breakdown of how to get there. Think of it this way:

- **PRD**: the destination — what you're building and why
- **Plan**: the journey — the ordered tasks that get you there

## Write a PRD

The most reliable way to write a PRD is to use a skill that asks the right questions and structures the output consistently. The `to-prd` skill from Matt Pocock's skills repo does exactly this — it converts your conversation context into a well-formed PRD.

[to-prd](https://github.com/mattpocock/skills/tree/main/to-prd)

## Write a tracer bullet plan

Without explicit guidance, an LLM will implement a feature horizontally — e.g. API first, then backend, then frontend — treating each layer as a complete task. This makes it hard to see a working slice of the feature until everything is done.

A better approach is **vertical, tracer bullet implementation**: pick the thinnest path through the full stack that proves the feature works end-to-end, ship that first, then iterate. One working slice early beats three half-finished layers late.
