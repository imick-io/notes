# Context Window & Quadratic Attention Complexity

We often hear that you should not bloat your context window — but why exactly?

The reason is the **self-attention mechanism** at the heart of transformer models. During attention, every token computes a relationship score against every other token, producing an n×n attention matrix. This means both compute and memory scale as **O(n²)** with respect to sequence length.

Double the context → 4× the work. At 64K tokens, a standard transformer requires roughly **250× more operations** than sparse alternatives. Even on an 80 GB GPU, training is limited to ~16K–23K tokens before memory is exhausted.

In practice this means:
- Larger context = exponentially slower inference
- Bloated prompts waste budget on tokens the model may barely "see" at the edges
- Keep context lean and relevant for both speed and quality

# Lost in the Middle

Even as models support larger context windows, they don't attend to all of it equally. Research by Liu et al. (2023) found a **U-shaped performance curve**: models reliably use information at the **beginning** and **end** of the context, but performance degrades significantly for content buried in the **middle**.

In one benchmark, GPT-3.5-Turbo's accuracy on a multi-document QA task dropped by over 20% when the relevant document was placed in the middle of the context vs. at the edges. In worst-case scenarios, performance fell below even the closed-book baseline (no context at all).

The cause is architectural: positional encodings (like RoPE) introduce a decay effect that biases attention toward early and recent tokens, and causal attention means earlier tokens participate in more attention operations overall.

**Practical implication:** put your most important content — instructions, key constraints, the actual question — at the start or end of your prompt. Don't bury it in the middle.

This also means you want to keep the middle as small as possible — which means keeping your overall context window as small as possible. A long conversation accumulates cruft: old reasoning, retried approaches, redundant tool output. All of that becomes dead weight in the middle. The best habit is to **start a new chat for every distinct task** rather than letting one conversation sprawl.

A useful rule of thumb: **stay below 100K tokens of the model's context limit**. Once you cross that threshold, the risk of the model losing track of earlier context grows meaningfully. Claude Code shows context usage in the status bar — treat it like a fuel gauge.

# LLMs Are Not Databases

You should not rely on an LLM as a source of truth. This is counterintuitive — the model was trained on an enormous slice of publicly available human knowledge — but the way it stores and retrieves that knowledge is fundamentally different from a database.

An LLM doesn't look anything up. It compresses training data into billions of numerical weights — a lossy encoding, not a searchable index. When you ask it a question, it doesn't retrieve a fact; it **predicts the most statistically likely next token**, one at a time, based on everything that came before. The objective during training was linguistic plausibility, not factual accuracy.

This is why **hallucinations are not a bug — they are an inherent property of the architecture**. Research confirms that training LLMs for predictive accuracy inevitably produces hallucinations, regardless of model design or data quality. A confidently stated falsehood and a correct answer can look identical from the model's perspective: both are high-probability token sequences.

**Practical implication:** treat LLM output like input from a knowledgeable colleague who occasionally misremembers things — useful for reasoning, drafting, and synthesis, but always verify facts, figures, and code against authoritative sources.

# LLMs Have No Persistent Memory

LLMs are **stateless by design**. Each conversation starts from zero — the model has no recollection of previous sessions, past decisions, or prior context unless that information is explicitly injected into the current context window. Any apparent memory is just conversation history that a developer or tool chose to pass along.

This is fundamentally different from how a human colleague works. A colleague builds a mental model of you over time: your preferences, past mistakes, ongoing projects. An LLM resets completely between chats.

There are patterns for working around this, each with trade-offs:

| Approach | How it works | Claude Code equivalent |
|---|---|---|
| **System prompt / config file** | Persistent instructions injected at the start of every session |  |
| **External memory store** | Key facts written to files and re-read as needed | Claude's built-in memory () |
| **RAG** | Relevant documents retrieved and injected into context at query time | Passing files or using  references |
| **Conversation buffer** | Full or summarised chat history re-fed each turn | What Claude Code does within a single session |

None of these give the LLM true memory — they are all just different strategies for getting the right information into the context window at the right time.

**Practical implication:** don't assume Claude remembers anything from a previous session. Put persistent context in and keep it current. 

# Navigate the limitations

Every limitation above stems from the same root: an LLM is a **statistical text predictor**, not a reasoning agent with a memory, a database, or unlimited attention.

Working effectively with Claude Code means designing around that reality:

- **Keep context small and focused** — quadratic attention makes bloat expensive, and the middle of your context is where attention goes to die
- **One task, one chat** — don't let conversations sprawl; reset early and often
- **Anchor critical information at the edges** — put key instructions at the top, key questions at the bottom
- **Never trust output blindly** — verify facts, figures, and generated code; hallucinations are architectural, not accidental
- **Compensate for statelessness deliberately** — use `CLAUDE.md` for persistent context rather than assuming Claude will remember

The models are remarkably capable within these constraints. The developers who get the most out of them are the ones who stop fighting the constraints and start designing for them. 

## Sub-agents

Because context is finite and attention degrades over long sequences, complex tasks benefit from being decomposed. Rather than attempting everything in one sprawling context window, an **orchestrator** agent breaks the work into discrete subtasks and delegates each to a **sub-agent** — a fresh model invocation with its own empty context window.

Each sub-agent focuses on a single well-scoped task, can spend its full token budget on that work, and returns a concise summary to the orchestrator when done. The orchestrator never sees the sub-agent's internal reasoning — only the result — keeping the orchestrator's own context lean.

Sub-agents can also run in **parallel**: the orchestrator dispatches multiple agents simultaneously and waits for all of them to report back before synthesising the results. This turns what would otherwise be a sequential, context-bloating monologue into a coordinated, efficient pipeline.

Because each sub-agent is a separate invocation, they can be configured independently — different system prompts, different models, different tool access — matching the agent to the nature of the subtask rather than forcing everything through one generalised configuration.

## Compaction

When the context window fills up, Claude Code will **automatically compact** the conversation — summarising the history into a condensed form so the session can continue without starting from scratch.

You can also trigger compaction manually with the `/compact` slash command. The key advantage of doing it yourself is that you can pass a short description: what you were working on and what you plan to do next. That framing helps the model produce a tighter, more relevant summary — one that preserves the details most likely to matter going forward rather than compressing everything uniformly.

Proactive compaction is most useful during long debugging sessions. Rather than losing track of what you've already tried as the context grows, you compact to lock in a summary of the current state and continue from a clean, smaller context.

**Practical implication:** if you're heads-down on a complex problem and notice the context gauge climbing, don't wait for auto-compaction — `/compact` with a one-line description of where you are and what you're trying to do next.
