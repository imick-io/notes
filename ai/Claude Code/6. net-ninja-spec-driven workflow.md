# Spec-Driven Workflow

## Overview

When working with AI on non-trivial features, asking it to implement something directly from a vague description often produces unfocused results. Writing a spec first gives the AI a precise target and keeps you in control of the direction. That's the core idea behind this workflow.

The workflow has four steps:

1. **Write a high-level spec** using the `/spec` skill — on a new branch
2. **Create a technical plan** using Plan mode
3. **Implement the feature** using a capable model with extended thinking
4. **Review the code** before committing

---

## Step 1 — Spec

The spec skills are custom skills that automate the boilerplate: they create a new Git branch, generate a structured spec file under `_specs/`, and print a summary of where everything was saved.

Available spec skills:
- [spec-simple](../skills/spec-simple.md) — text-only feature description
- [spec-figma](../skills/spec-figma.md) — includes a Figma component link for design reference
- [spec-figma-agent](../skills/spec-figma-agent.md) — same as spec-figma but uses an agent to extract design details from Figma automatically

### Spec template

Each skill generates a spec file using a shared template. Define this template at `_specs/template.md` in your project:

```md
# Spec for <feature-name>

branch: claude/feature/<feature-name>  
figma_component (if used): <figma-component-name>

## Summary
...

## Functional Requirements
- ...

## Figma Design Reference (only if referenced)
- File: ...
- Component name: ...
- Key visual constraints: ...

## Possible Edge Cases
- ...

## Acceptance Criteria
- ...

## Open Questions
- ...

## Testing Guidelines
Create a test file(s) in the `./tests` folder for the new feature, and create meaningful tests for the following cases, without going too heavy:
- ...
```

### Usage

Run `/spec` followed by a short feature description. For example:

```
/spec Let's spec the authentication forms on the /login and /signup pages. Both need email and password fields, a password visibility toggle, and a submit button. For now, the forms should only log to the console on submit.
```

Once the spec file is generated, review the **Open Questions** section and fill in your answers. For production work, take the time to read through the whole spec and adjust anything that doesn't match your intent before moving on.

---

## Step 2 — Plan

With the spec in place, use Plan mode to turn it into a concrete technical plan. This step is valuable because Claude Code reads the current codebase to produce the plan, giving it real context about your project's structure and conventions — not just the spec in isolation. It also produces a reviewable artifact you can inspect and adjust before any code is written.

To enter Plan mode, press `SHIFT+TAB` to cycle through the input modes until **Plan** is selected.

Then ask:

```
Plan the feature described in this spec and store it in the _plans folder.
```

Add the spec file as context using `@` followed by the filename, or by pressing `Cmd+Option+K` to open the file picker.

When the plan is generated, **do not proceed immediately** — review it first. Make sure the approach, file locations, and scope match what you intended.

---

## Step 3 — Implement

Before starting the implementation, commit all your current work. This gives you a clean rollback point if you don't like what the AI produces.

For implementation, use the most capable model available. Switch models with the `/model` command. It's also worth enabling extended thinking, which causes the model to reason through the problem more carefully before writing code. You can toggle it with `Option+T`, or enable it persistently via `/config` under the thinking mode setting.

---

## Step 4 — Code Review

When the implementation is complete, review the diff before committing. You can use the `/review` skill if you have it, or just read through the changes manually. Look for:

- Logic that doesn't match the spec or acceptance criteria
- Missing edge case handling
- Anything that looks unintended or outside the scope of the feature

Once you're satisfied, commit and move on.
